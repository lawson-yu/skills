# 问题追踪器集成仅限主流工具

`setup-matt-pocock-skills` 仅为**主流**问题追踪器提供一流支持。为小众、全新或单一厂商实验性追踪器添加支持的请求不在范围内。

## 为什么这不在范围内

每个问题追踪器后端都会在技能中硬编码一种 CLI 形态（命令、标志、输出解析）。每新增一个后端，都会带来永久性的维护面 —— 它必须随着工具的 CLI 演进持续可用，并且必须持续在 `/to-prd`、`/to-issues`、`/triage` 及相关命令上进行测试。只有当某个追踪器确实被相当比例的用户使用时，这个成本才值得承担。

“主流”是一个判断问题，而不是数字门槛：

- GitHub、GitLab 和 Backlog.md 属于我们会认为是主流的工具 —— 广为人知、广泛使用，并且早已过了实验阶段。
- 一个面向 agent 的全新工具，即使设计再有趣，如果只有几百个 GitHub stars，也不算主流。

在做判断时，stars、项目年龄和下载量都是有用信号，但它们都不是规则本身。真正的规则是：一个典型工程师是否会认得这个工具，并且有可能为自己的团队选择它？

针对非主流追踪器的替代出口已经存在：

- `local markdown` 用于轻量级仓库内追踪。
- `other/custom` 用于希望自行接入的用户。

这两种方式都不要求核心技能了解该具体工具。
