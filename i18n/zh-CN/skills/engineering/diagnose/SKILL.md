# 诊断

用于处理疑难 bug 的纪律。仅在有明确理由时才可跳过阶段。

在探索代码库时，使用项目的领域术语表来建立相关模块的清晰心智模型，并查看你所涉及区域内的 ADR。

## 阶段 1 — 建立反馈回路

**这就是这个 skill。** 其他一切都只是机械步骤。如果你有一个快速、确定性、可由 agent 运行的 bug 通过/失败信号，你就能找到根因——二分查找、假设检验和插桩都只是消费这个信号。如果你没有，无论盯着代码看多久都无济于事。

在这里投入不成比例的精力。**要激进。要有创造性。拒绝放弃。**

### 构建反馈回路的方法——按大致顺序尝试

1. 在能够触达 bug 的接缝处编写**失败测试**——unit、integration、e2e 均可。
2. 对运行中的开发服务器使用 **Curl / HTTP 脚本**。
3. 使用带夹具输入的 **CLI 调用**，并将 stdout 与已知正确快照做 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，并对 DOM/console/network 断言。
5. **回放已捕获的 trace。** 将真实网络请求 / payload / 事件日志保存到磁盘；在隔离环境中通过该代码路径回放。
6. **一次性 harness。** 启动系统的最小子集（一个服务，mock 依赖），通过一次函数调用触发 bug 代码路径。
7. **性质 / fuzz 循环。** 如果 bug 是“有时输出错误”，运行 1000 个随机输入并寻找失败模式。
8. **二分 harness。** 如果 bug 出现在两个已知状态（commit、数据集、版本）之间，自动化“在状态 X 启动、检查、重复”，以便 `git bisect run`。
9. **差分循环。** 对同一输入运行旧版本与新版本（或两套配置），并 diff 输出。
10. **HITL bash 脚本。** 最后手段。如果必须人工点击，使用 `scripts/hitl-loop.template.sh` 来驱动 _them_，使循环仍然结构化。捕获的输出再反馈给你。

构建正确的反馈回路，bug 就完成了 90%。

### 迭代反馈回路本身

把这个回路当作产品。一旦你有了 _一个_ 回路，就问：

- 我能让它更快吗？（缓存环境、跳过无关初始化、缩小测试范围。）
- 我能让信号更锐利吗？（对具体症状断言，而不是“没有崩溃”。）
- 我能让它更确定吗？（固定时间、固定 RNG seed、隔离文件系统、冻结网络。）

一个 30 秒且易波动的回路几乎不比没有好多少。一个 2 秒且确定性的回路就是调试超能力。

### 非确定性 bug

目标不是“干净复现”，而是**更高的复现率**。将触发过程循环 100 次、并行化、加压测、收窄时序窗口、注入 sleep。50% 波动率的 bug 是可调试的；1% 不是——持续提高复现率，直到可调试为止。

### 当你确实无法构建回路时

停下来并明确说明。列出你尝试过的内容。向用户请求：(a) 可复现该问题的环境访问权限，(b) 已捕获的工件（HAR 文件、日志转储、core dump、带时间戳的录屏），或 (c) 允许添加临时生产环境插桩。**不要**在没有回路的情况下继续提出假设。

在你拥有一个你信任的回路之前，不要进入阶段 2。

## 阶段 2 — 复现

运行回路。观察 bug 出现。

确认：

- [ ] 回路产生的是**用户描述**的失败模式——而不是恰好邻近的另一种失败。修错 bug = 错修复。
- [ ] 该失败可在多次运行中复现（或针对非确定性 bug，复现率足够高，能够用于调试）。
- [ ] 你已捕获确切症状（错误信息、错误输出、慢耗时），以便后续阶段验证修复确实命中问题。

在复现 bug 之前，不要继续。

## 阶段 3 — 提出假设

在测试任何假设之前，先生成**3–5 个有排序的假设**。只提出单一假设会锚定在第一个看似合理的想法上。

每个假设都必须**可证伪**：说明它给出的可验证预测。

> 格式：“如果 <X> 是原因，那么 <改变 Y> 会让 bug 消失 / <改变 Z> 会让它更严重。”

如果你说不出预测，这个假设只是感觉——丢弃或继续打磨。

**在测试前向用户展示排序后的列表。** 他们通常有领域知识可以立刻重排（“我们刚部署了 #3 的变更”），或知道哪些假设已被排除。这是低成本检查点，能大量省时。不要被它阻塞——如果用户 AFK，就按你的排序继续。

## 阶段 4 — 插桩

每个探针都必须映射到阶段 3 的特定预测。**每次只改变一个变量。**

工具优先级：

1. 如果环境支持，优先用 **Debugger / REPL inspection**。一个断点胜过十条日志。
2. 在能区分假设的边界处添加**有针对性的日志**。
3. 绝不“全部都打日志再 grep”。

**给每条调试日志加唯一前缀**，例如 `[DEBUG-a4f2]`。这样结束时清理只需一次 grep。无标签日志会残留；有标签日志必须清除。

**性能分支。** 对性能回归而言，日志通常是错误工具。应改为：建立基线测量（计时 harness、`performance.now()`、profiler、query plan），然后做二分。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在修复前先写回归测试——但仅当存在**正确接缝**时。

正确接缝是指：测试在调用点能够复现**真实 bug 模式**。如果唯一可用接缝过浅（例如 bug 需要多个调用方但只有单调用方测试，或 unit 测试无法复现触发 bug 的调用链），在该处写回归测试会制造虚假信心。

**如果不存在正确接缝，这本身就是结论。** 记录它。代码库架构正在阻止该 bug 被有效锁定。将此标记为下一阶段事项。

如果存在正确接缝：

1. 将最小化复现转成该接缝上的失败测试。
2. 观察其失败。
3. 应用修复。
4. 观察其通过。
5. 用原始（未最小化）场景重新运行阶段 1 的反馈回路。

## 阶段 6 — 清理 + 复盘

在宣布完成前必须满足：

- [ ] 原始复现不再出现（重新运行阶段 1 回路）
- [ ] 回归测试通过（或已记录不存在接缝）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（`grep` 该前缀）
- [ ] 一次性原型已删除（或移动到明确标注的调试位置）
- [ ] 已在 commit / PR 信息中说明最终成立的假设——让下一位调试者能够学习

**然后问：本可以如何预防这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、调用方纠缠、隐藏耦合），把具体信息移交给 `/improve-codebase-architecture` skill。请在修复完成**之后**再给出建议，而不是之前——此时你的信息比刚开始时更充分。
