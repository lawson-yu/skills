# ADR 格式

ADRs 存放在 `docs/adr/` 中，并使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

惰性创建 `docs/adr/` 目录——仅在需要第一个 ADR 时创建。

## 模板

```md
# {Short title of the decision}

{1-3 sentences: what's the context, what did we decide, and why.}
```

就是这样。ADR 可以只有一个段落。价值在于记录*确实*做出了某个决策以及*为什么*，而不是把各个章节填满。

## 可选章节

仅当它们确实能增加价值时才包含。大多数 ADR 都不需要它们。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时很有用
- **Considered Options** ——仅当被拒绝的备选方案值得记住时
- **Consequences** ——仅当需要指出不明显的下游影响时

## 编号

扫描 `docs/adr/` 中现有的最大编号，并在其基础上加一。

## 何时建议使用 ADR

以下三点必须同时满足：

1. **难以逆转** ——之后改变主意的成本是有意义的
2. **脱离上下文会令人意外** ——未来读者看到代码会想“他们为什么要这样做？”
3. **真实权衡的结果** ——确实存在可选方案，并且你基于具体原因做了选择

如果一个决策很容易逆转，就跳过——你之后大概率会直接改掉。如果它并不令人意外，就没人会疑惑为什么。如果没有真实备选方案，除了“我们做了显而易见的事”之外也没什么可记录的。

### 哪些情况算

- **架构形态。** “我们使用 monorepo。” “写模型采用 event-sourced，读模型投影到 Postgres。”
- **上下文之间的集成模式。** “Ordering 和 Billing 通过 domain events 通信，而不是同步 HTTP。”
- **会产生锁定效应的技术选择。** 数据库、消息总线、认证提供商、部署目标。不是每个库——只记录那些替换起来需要一个季度的。
- **边界和范围决策。** “Customer 数据由 Customer context 拥有；其他 context 仅通过 ID 引用。” 明确的“不做什么”与“做什么”同样有价值。
- **有意偏离显而易见路径。** “我们使用手写 SQL 而不是 ORM，因为 X。” 任何合理读者会默认相反的选择。这能防止下一个工程师把有意为之的设计“修正”掉。
- **代码中看不见的约束。** “由于合规要求，我们不能使用 AWS。” “由于合作方 API 合同，响应时间必须低于 200ms。”
- **当拒绝理由并不明显时，记录被拒绝的备选方案。** 如果你考虑过 GraphQL，因一些微妙原因选择了 REST，就把它记下来——否则六个月后还会有人再次提议 GraphQL。
