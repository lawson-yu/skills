# Triage

通过一套小型的 triage 角色状态机，推动项目 issue 跟踪器中的 issue 流转。

在 triage 期间发布到 issue 跟踪器的每条评论或 issue **都必须**以以下免责声明开头：

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 如何编写可长期使用的 agent 简报
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` 知识库的工作方式

## 角色

两个**类别**角色：

- `bug` — 某些内容已损坏
- `enhancement` — 新功能或改进

五个**状态**角色：

- `needs-triage` — 维护者需要评估
- `needs-info` — 正在等待报告者提供更多信息
- `ready-for-agent` — 规格完整，可交由 AFK agent
- `ready-for-human` — 需要人工实现
- `wontfix` — 不会处理

每个完成 triage 的 issue 应该恰好包含一个类别角色和一个状态角色。如果状态角色冲突，先标记并询问维护者，再进行其他操作。

这些是规范角色名称——issue 跟踪器中实际使用的标签字符串可能不同。映射应已提供给你——如果没有，请运行 `/setup-matt-pocock-skills`。

状态流转：未标记的 issue 通常先进入 `needs-triage`；然后可流转到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后回到 `needs-triage`。维护者可随时覆盖——若发现看起来异常的流转，请先标记并询问后再继续。

## 调用方式

维护者调用 `/triage` 并用自然语言描述他们想做的事。你需要理解请求并执行。示例：

- “Show me anything that needs my attention”
- “Let's look at #42”
- “Move #42 to ready-for-agent”
- “What's ready for agents to pick up?”

## 展示需要关注的内容

查询 issue 跟踪器，并按从旧到新展示三个分组：

1. **未标记** — 从未做过 triage。
2. **`needs-triage`** — 正在评估中。
3. **`needs-info` 且在上次 triage 记录后报告者有新活动** — 需要重新评估。

展示计数以及每个 issue 的一行摘要。让维护者选择。

## 对特定 issue 进行 triage

1. **收集上下文。** 阅读完整 issue（正文、评论、标签、报告者、日期）。解析任何先前的 triage 记录，避免重复询问已解决的问题。使用项目的领域术语表探索代码库，并遵循相关区域的 ADR。阅读 `.out-of-scope/*.md`，并指出任何与该 issue 相似的历史拒绝案例。

2. **给出建议。** 告诉维护者你建议的类别和状态，并给出理由，再补充与该 issue 相关的简要代码库摘要。等待指示。

3. **复现（仅 bug）。** 在任何 grilling 之前，先尝试复现：阅读报告者步骤，追踪相关代码，运行测试或命令。报告结果——复现成功并给出代码路径、复现失败，或细节不足（这是强烈的 `needs-info` 信号）。已确认复现会让 agent 简报更有力。

4. **Grill（如需要）。** 如果 issue 需要补充细化，运行一次 `/grill-with-docs` 会话。

5. **应用结果：**
   - `ready-for-agent` — 发布一条 agent 简报评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` — 使用与 agent 简报相同的结构，但说明为何无法委派（判断决策、外部访问、设计决策、手工测试）。
   - `needs-info` — 发布 triage 记录（见下方模板）。
   - `wontfix`（bug）— 礼貌说明原因，然后关闭。
   - `wontfix`（enhancement）— 写入 `.out-of-scope/`，在评论中附上链接，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` — 应用该角色。如有部分进展，可选地发表评论。

## 快速状态覆盖

如果维护者说“move #42 to ready-for-agent”，请信任并直接应用该角色。先确认你将要执行的操作（角色变更、评论、关闭），然后执行。跳过 grilling。如果在没有 grilling 会话的情况下移动到 `ready-for-agent`，请询问他们是否要编写 agent 简报。

## Needs-info 模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

将 grilling 期间已解决的所有内容记录在“established so far”下，避免工作丢失。问题必须具体且可执行，不能写成“please provide more info”。

## 恢复之前的会话

如果该 issue 中已有先前 triage 记录，请先阅读，检查报告者是否已回答任何待解决问题，并在继续前给出更新后的整体情况。不要重复询问已解决的问题。
