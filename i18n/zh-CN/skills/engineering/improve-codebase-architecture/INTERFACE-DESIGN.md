# 接口设计

当用户希望为选定的深化候选探索替代接口时，使用这个并行子代理模式。基于 “Design It Twice”（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇——**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 界定问题空间

在生成子代理之前，先为选定候选编写一段面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖的依赖项，以及这些依赖所属的类别（见 [DEEPENING.md](DEEPENING.md)）
- 一个用于锚定约束的粗略示意代码草图——不是提案，只是将约束具体化的一种方式

将这段内容展示给用户，然后立刻进入第 2 步。用户在阅读和思考时，子代理并行工作。

### 2. 生成子代理

使用 Agent 工具并行生成 3 个以上子代理。每个子代理都必须为深化后的模块产出一个**截然不同**的接口。

为每个子代理提供独立的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、seam 后面是什么）。该简报独立于第 1 步中面向用户的问题空间说明。给每个代理指定不同的设计约束：

- Agent 1：“最小化接口——目标是最多 1–3 个入口点。最大化每个入口点的 leverage。”
- Agent 2：“最大化灵活性——支持多种用例和扩展。”
- Agent 3：“为最常见的调用方优化——让默认场景变得极其简单。”
- Agent 4（如适用）：“围绕 ports & adapters 设计跨-seam 依赖。”

在简报中同时包含 [LANGUAGE.md](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，以便每个子代理在命名时与架构语言和项目领域语言保持一致。

每个子代理输出：

1. 接口（types、methods、params——以及 invariants、ordering、error modes）
2. 用法示例，展示调用方如何使用它
3. 实现隐藏在 seam 后面的内容
4. 依赖策略和 adapters（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里 leverage 高，哪里较薄弱

### 3. 展示与比较

按顺序展示各个设计，让用户能逐一吸收，然后用 prose 进行比较。对比维度为 **depth**（接口层的 leverage）、**locality**（变更集中的位置）和 **seam placement**。

比较后给出你自己的建议：你认为哪个设计最强，以及原因。如果不同设计中的元素适合组合，提出一个混合方案。要有明确立场——用户想要的是有力判断，而不是选项菜单。
