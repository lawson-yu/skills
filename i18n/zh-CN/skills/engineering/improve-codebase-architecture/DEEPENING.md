# 深化

在给定依赖关系的前提下，如何安全地深化一组浅模块。假设你已掌握 [LANGUAGE.md](LANGUAGE.md) 中的术语——**module**、**interface**、**seam**、**adapter**。

## 依赖类别

在评估一个候选深化对象时，先对其依赖进行分类。类别决定了深化后的模块如何跨越其 seam 进行测试。

### 1. 进程内

纯计算、内存状态、无 I/O。始终可深化——合并模块并通过新接口直接测试。无需 adapter。

### 2. 本地可替代

在本地有测试替身的依赖（Postgres 的 PGLite、内存文件系统）。如果存在替身即可深化。深化后的模块在测试套件中使用该替身运行测试。seam 是内部的；模块外部接口上不需要 port。

### 3. 远程但自有（Ports & Adapters）

跨网络边界的自有服务（微服务、内部 API）。在 seam 处定义 **port**（接口）。深模块拥有逻辑；传输通过注入的 **adapter** 提供。测试使用内存 adapter。生产使用 HTTP/gRPC/queue adapter。

推荐表述：*"在 seam 处定义一个 port，为生产实现 HTTP adapter、为测试实现内存 adapter，这样即便跨网络部署，逻辑仍位于一个深模块中。"*

### 4. 真正外部（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的 port；测试提供 mock adapter。

## Seam 纪律

- **一个 adapter 表示假想 seam。两个 adapter 才表示真实 seam。** 除非至少有两个 adapter 的合理性（通常是生产 + 测试），否则不要引入 port。单 adapter 的 seam 只是间接层。
- **内部 seam 与外部 seam。** 深模块既可以有内部 seam（其实现私有、供自身测试使用），也可以在其接口处有外部 seam。不要仅仅因为测试使用了内部 seam 就把它通过接口暴露出去。

## 测试策略：替换，而非叠加

- 一旦在深化模块的接口处建立了测试，原先浅模块上的单元测试就会变成浪费——删除它们。
- 在深化模块的接口处编写新测试。**接口就是测试表面**。
- 测试应通过接口对可观察结果进行断言，而不是内部状态。
- 测试应在内部重构后仍然有效——它们描述的是行为，而不是实现。如果实现一变测试就必须改，说明它测试越过了接口。
