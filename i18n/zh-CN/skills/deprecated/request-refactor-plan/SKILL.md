---

当用户想要创建重构请求时，将调用此技能。你应按以下步骤进行。如果你认为某些步骤没有必要，可以跳过。

1. 请用户详细、充分地描述他们想要解决的问题，以及任何可能的解决思路。

2. 探索仓库以验证他们的陈述，并理解代码库的当前状态。

3. 询问他们是否考虑过其他选项，并向他们展示其他选项。

4. 与用户就实现方案进行访谈。要非常细致和全面。

5. 明确实现的精确范围。厘清你计划修改的内容以及不计划修改的内容。

6. 在代码库中检查该区域的测试覆盖情况。如果测试覆盖不足，询问用户的测试计划是什么。

7. 将实现拆分为微小提交的计划。记住 Martin Fowler 的建议：“让每一步重构都尽可能小，这样你始终可以看到程序在工作。”

8. 使用以下 issue 描述模板创建一个 GitHub issue，包含重构计划：

<refactor-plan-template>

## 问题陈述

开发者所面临的问题，从开发者视角描述。

## 解决方案

问题的解决方案，从开发者视角描述。

## 提交计划

一个**长篇且详细**的实现计划。使用自然语言撰写，并将实现拆分为尽可能细小的提交。每个提交都应让代码库保持可工作的状态。

## 决策文档

列出已做出的实现决策。可以包括：

- 将构建/修改的模块
- 这些模块将被修改的接口
- 来自开发者的技术澄清
- 架构决策
- Schema 变更
- API 合约
- 特定交互

不要包含具体文件路径或代码片段。它们可能很快就会过时。

## 测试决策

列出已做出的测试决策。包括：

- 对“什么是好测试”的描述（只测试外部行为，而非实现细节）
- 将被测试的模块
- 测试的先例（即代码库中类似类型的测试）

## 范围之外

描述本次重构不在范围内的事项。

## 进一步说明（可选）

关于此次重构的任何进一步说明。

</refactor-plan-template>
